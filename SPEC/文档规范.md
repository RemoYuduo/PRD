# PRD文档规范

本文档基于《PRD-擎天运维平台-服务树系统V1.0.0-服务信息管理-20190722》分析提炼，用于指导后续PRD文档的编写。

---

## 一、文档命名规范

### 1.1 文件命名格式

```
PRD-{产品名称}-{模块名称}V{版本号}-{功能名称}-{日期}.md
```

**命名要素说明：**

| 要素 | 格式 | 示例 |
| --- | --- | --- |
| 产品名称 | 中文名 | 擎天运维平台 |
| 模块名称 | 中文名 | 服务树系统 |
| 版本号 | X.Y.Z | V1.0.0 |
| 功能名称 | 中文名 | 服务信息管理 |
| 日期 | YYYYMMDD | 20190722 |

### 1.2 版本号规则

- **主版本号(X)**：重大功能重构或架构变更
- **次版本号(Y)**：新增功能模块
- **修订号(Z)**：Bug修复或小幅优化

---

## 二、文档头部信息

### 2.1 基本信息表

每份PRD文档开头必须包含以下元信息表格：

| 字段 | 说明 | 必填 |
| --- | --- | --- |
| PRD审核人 | 负责审核PRD的人员 | ✓ |
| 重要性 | P0/P1/P2/P3（P0最高） | ✓ |
| 紧迫性 | 高/中/低 | ✓ |
| PRD拟制人 | 文档编写者 | ✓ |
| PRD提交日期 | 首次提交日期 | ✓ |
| 需求变更控制时间点 | 变更截止时间 | 选填 |

### 2.2 修改记录表

文档头部需包含PRD修改记录表，记录每次变更：

| 字段 | 说明 |
| --- | --- |
| PRD更新时间 | 变更日期 |
| 变更内容 | 简要描述变更内容 |
| 变更提出人 | 提出变更的人员 |
| 变更理由 | 变更原因（可选） |

## 三、文档结构规范

### 3.1 标准章节结构

PRD文档应包含以下标准章节：

```
1. 背景描述
2. 用户角色与权限描述
3. 产品概述
   3.1 总体流程
4. 产品需求
   4.x 功能模块（可有多个）
       4.x.y 具体功能点
5. 非功能性需求
```

### 3.2 各章节内容要求

#### 1. 背景描述
- **目的**：说明为什么要做这个产品/功能
- **内容**：描述当前痛点、业务需求、未来规划
- **篇幅**：2-5段，简明扼要

#### 2. 用户角色与权限描述
- **目的**：明确产品涉及的用户类型及其权限
- **内容**：
  - 角色分类（系统级角色、业务级角色等）
  - 每个角色的描述
  - 每个角色的默认权限
- **格式**：使用表格清晰列出

**示例格式：**

| 角色 | 描述 | 默认权限 |
| --- | --- | --- |
| Admin | 平台管理员 | 拥有全部前后台管理权限 |
| 普通用户 | 普通使用者 | 基础查看权限 |

#### 3. 产品概述
- **目的**：从宏观层面描述产品架构和核心流程
- **内容**：
  - 总体流程图（可使用流程图工具或文字描述）
  - 核心业务流程说明
  - 模块关系说明

#### 4. 产品需求
- **目的**：详细描述每个功能点
- **内容**：按模块组织，每个功能点应包含：
  - 功能描述
  - 操作流程
  - 界面说明（可附原型图）
  - 业务规则
  - 异常处理

#### 5. 非功能性需求
- **目的**：描述功能之外的系统要求
- **内容**：
  - 性能要求（响应时间、并发数等）
  - 安全要求
  - 兼容性要求
  - 可用性要求
  - 可维护性要求

## 四、功能描述规范

### 4.1 功能点描述模板

每个功能点应按以下结构描述：

```markdown
### X.Y.Z 功能名称

**功能描述**：简要说明该功能的作用

**前置条件**：执行该功能前需要满足的条件

**操作流程**：
1. 步骤一
2. 步骤二
3. ...

**界面说明**：
- 界面元素描述
- 交互说明
- 原型图（如有）

**业务规则**：
- 规则1
- 规则2

**异常处理**：
- 异常情况1 → 处理方式
- 异常情况2 → 处理方式
```

### 4.2 字段说明格式

当涉及表单字段时，使用表格说明：

| 字段名 | 类型 | 必填 | 可编辑 | 说明 |
| --- | --- | --- | --- | --- |
| 服务名称 | 文本 | ✓ | × | 系统自动生成，不可修改 |
| 服务简介 | 文本 | ✓ | ✓ | 限500字以内 |

### 4.3 权限控制说明

功能涉及权限时，需明确说明：
- 哪些角色可以执行该操作
- 不同角色的操作差异
- 权限继承规则

---

## 五、表格与列表规范

### 5.1 分页说明
描述列表页面时需说明：
- 每页显示条数
- 默认排序规则
- 搜索范围（当前页/全局）

### 5.2 搜索功能说明
- 搜索支持的字段
- 模糊匹配/精确匹配
- 大小写敏感性
- 实时搜索/回车搜索

### 5.3 批量操作说明
- 支持的批量操作类型
- 选中状态的按钮变化
- 确认提示内容

---

## 六、状态与流程规范

### 6.1 按钮状态说明

需明确说明按钮在不同状态下的表现：

| 状态 | 按钮表现 | 触发条件 |
| --- | --- | --- |
| 可用 | 正常颜色可点击 | 满足操作前置条件 |
| 禁用 | 灰色不可点击 | 不满足操作前置条件 |
| 加载中 | Loading状态 | 操作执行中 |

### 6.2 操作确认说明

删除、修改等重要操作需要说明确认流程：
- 确认对话框内容
- 展示哪些信息供用户确认
- 确认/取消后的行为

### 6.3 功能预留说明

对于当前版本不实现的功能：
- 明确标注"当前版本暂不涉及"
- 说明界面元素状态（如：置灰）
- 简要描述后续规划

## 七、写作风格规范

### 7.1 语言要求
- 使用陈述句，避免疑问句
- 表述清晰准确，避免歧义
- 使用统一的术语

### 7.2 动词使用
- 点击、选择、输入、提交、确认、取消
- 展示、渲染、跳转、弹出
- 保存、删除、修改、新增

### 7.3 条件表述
- 使用"如果...则..."结构
- 条件和结果要明确对应
- 复杂条件使用列表形式

**示例：**
```
如果用户没有选中任何机器：
- "删除机器"按钮为灰色不可点击

如果选中了至少一台机器：
- "删除按钮"可点击
- 点击后弹出确认对话框
```

---

## 八、版本管理规范

### 8.1 版本迭代说明

当功能分多版本实现时：
- 明确当前版本范围
- 标注后续版本规划
- 使用"V1.0"、"后续版本"等标识

### 8.2 变更记录要求

每次变更需要记录：
- 变更时间
- 变更内容摘要
- 变更提出人
- 变更理由（重大变更必填）

## 九、附录

### 9.1 常用缩写

| 缩写 | 全称 | 说明 |
| --- | --- | --- |
| PRD | Product Requirements Document | 产品需求文档 |
| RD | Research & Development | 研发人员 |
| QA | Quality Assurance | 质量保障/测试人员 |
| OP | Operations | 运维人员 |
| CAS | Central Authentication Service | 统一认证服务 |
| LDAP | Lightweight Directory Access Protocol | 轻量目录访问协议 |

### 9.2 重要性级别定义

| 级别 | 说明 |
| --- | --- |
| P0 | 核心功能，必须实现 |
| P1 | 重要功能，优先实现 |
| P2 | 一般功能，资源允许时实现 |
| P3 | 锦上添花，后续考虑 |

### 9.3 紧迫性级别定义

| 级别 | 说明 |
| --- | --- |
| 高 | 需立即启动，限期完成 |
| 中 | 按计划推进 |
| 低 | 可延后处理 |
